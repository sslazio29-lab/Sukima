# Sukima 引き継ぎ書（3択クイズアプリ・新規プロジェクト）v2

新規チャットの冒頭で、まずこの引き継ぎ書を読んでから作業を始めること。

> **このプロジェクトについて**
> Sukima は、Voice Study（音声教材作成アプリ）とは**別の新規プロジェクト**。
> - **Voice Study**＝因果で固める論述対策。「聞いて理解する」音声教材。voice_study + AirDrop の経路で今後も継続（Sukima には載せない）。
> - **Sukima**＝暗記物を高速に回す3択アプリ。「解いて定着させる」。
> 脳の使い方が違うので住み分けできる。両者のフォルダ・記録・セッションは混ぜないこと。
>
> ※この v2 は、構想初回セッションの全合意をまとめた確定版。「音声も配信する」前提だった旧 v1 は破棄。

---

## 0. このプロジェクトの一言まとめ

**Claude が作った3択問題と解説を、受験生（依頼者の子。京大・阪大志望）が iPhone でサクサク解いていく、全科目対応の暗記特化アプリ。**

- **パッと始めてパッと終われる**のが最優先の体験。
- 受験生はすでに自宅で自分で勉強している。**Sukima は勉強時間の配分を変えない「スキマの補助」に徹する**（最重要の歯止め）。

---

## 1. 絶対に守る歯止め（機能を盛らない）

このアプリが失敗する最大の原因は「複雑になりすぎること」。以下を厳守する。

- **やることは1つだけ**：3択を解く。付随して解答後に〇×と解説が出る。これ以外を作らない。
- **二本柱を一本に絞ったことが最大の強み**：構想当初は「聞き流し＋クイズ」だったが、音声は Voice Study + AirDrop が既に稼働・好評のため、Sukima は **3択クイズに全振り**。作るものが半分になり、かつ全科目に広がった。
- **やり込ませる仕掛けを入れない**：学習時間記録・成績グラフ・連続日数・ランキング等は入れない。熱心な子ほど時間を奪われ、勉強時間の配分を変えてしまうため。
- **シンプルさが複雑化への最大の防御**。迷ったら「足す」より「足さない」。

---

## 2. 全科目に効く理由（3択という器）

中身が何であれ「問題文・選択肢3つ・正解・解説」という同じ器に収まる。だから全科目に応用できる。

- **倫理・日本史**：正誤・組合せ問題を3択化（Claude が量産）。共通テスト型。
- **英単語**：語義・用法の3択（Claude が量産）。
- **数学**：計算させない。「この問題、最初の一手はどれ？」と**解法・方針を3択で選ばせる**。図を最小化でき、受験生の弱点（多段階の方針判断）に直撃。解説が短い学びになり、音声教材の役割を一部肩代わりする。
- **化学・生物・古文・漢文**：同じ器でそのまま拡張可能。

### 作問担当の区別（重要）
- **数学**＝受験生が実際に間違えた問題を依頼者が少し改題したオリジナル。弱点直撃。
- **暗記科目・英単語**＝**Claude（LLM）が定期的に作問・配信してよい**。クイズは新規に作る成果物なので、Voice Study の「内容改変禁止」ルールの対象外＝作ってよい領域。

---

## 3. 出題のテンポと「克服」の仕組み（確定）

「間違いは克服させたい」と「パッと始めてパッと終わりたい」を両立させる、依頼者考案の優れた仕組み。

- **1セッションの問題数は受験生が選ぶ**（固定しない）。電車1区間・休み時間にちょうど終わる量を自分で調整できる。
- **各問題は「未回答／回答済」の2状態**を持つ。
  - 出題は**未回答プールから引く**。
  - **正解したら「回答済」になり、もう出ない。**
  - **間違えたら「未回答」のまま残る**ので、また引かれる。
- これにより、普通に解き進めるだけで**苦手な問題だけが自然にしつこく出てくる**。「復習モード」という別画面を作らずに克服が実現する（独立画面は作らない。出題ロジックに溶け込ませる）。
- **全問が「回答済」になったら、リセットボタンで全問を未回答に戻せる**。また一周できる。
- **出題順はシャッフル**：間違えた問題が直後に出ると「答えを覚えているだけ」になるため、少し間隔をあけて再会させる。「理解して正解」と「記憶が新鮮なだけ」を区別するための細部。

### 画面（極限までシンプル）
1. 科目を選ぶ → 単元を選ぶ
2. 問題数を選ぶ
3. 問題が出る → 選択肢を3つタップで選ぶ
4. 即座に〇× → 解説 → 次へ
5. （全問回答済なら）リセットできる

凝った採点・タイマー・点数集計は作らない。「サクサク次へ」のテンポだけを追求する。

## 4. 技術構成（GitHub だけ・Cloudflare不要）

音声をやめ「問題JSONを配るだけ」にした結果、**純粋な静的アプリ**になった。GitHub Pages だけで完結する。

- **GitHub Pages 一本**：画面・判定ロジック・問題データ(JSON)を1リポジトリに入れて公開。
- **サーバー処理ゼロ**：「正解はどれか」「解説を出す」「回答済の記録」は**すべて受験生の iPhone のブラウザ内で完結**。問題JSONを一度読み込めば、あとはほぼ通信しない。
- **Cloudflare・Workers・KV・D1・R2 はすべて不要**。前回 Cloudflare で苦労した複雑さ（binding設定、デプロイの無言失敗など）をまるごと回避できる。
- **問題追加＝リポジトリに JSON を足して push するだけ**。依頼者の既存ワークフロー（コミット・プッシュ）にそのまま乗る。Claude Code / Codex もファイル編集と git 操作だけで完結し、事故りにくい。
- **「回答済」記録の保存**：iPhone ブラウザ内に残せる軽いデータ（数KB）。音声と違い容量問題は無関係。静的アプリのまま実現可能。
  - ※実装の保存先（localStorage 等）の選定は、Artifact ではなく独立 Web アプリなので通常のブラウザ保存APIが使える。最新の挙動は実装直前に確認すること。

### オフライン・通信（前回の悩みが消滅）
**音声を配信しないので、3GB制約・iPhoneオフライン保存の悩みがほぼ消滅。** 問題と解説はテキストで軽量。旧構想で最大の難所だった「IndexedDB必須・Cache API 50MB上限・今日のぶんだけ持ち出す」は**すべて不要**になった。これは全振りの大きな副産物。

### リポジトリ公開範囲（要確認）
- **Public にすれば完全無料**だが、問題と解説が世界中から見える。家庭の学習教材なので実害はほぼないと想定。
- 気になるなら **Private リポジトリ＋GitHub Pages**。無料プランで Private リポジトリの Pages 公開が可能かは、作る直前に最新を確認すること。

---

## 5. データの形（問題追加の手軽さが心臓部）

全振りした以上、アプリの価値はほぼ「問題をいかに簡単に増やせるか」に集約される。

- **1ファイル＝1単元の問題セット**（JSON）。命名規則は Voice Study を踏襲（例：`日本史_室町時代08.json`）。フォルダに置くだけでアプリが認識する設計にする。
- **1問の構造**：`問題文 / 選択肢3つ / 正解番号 / 解説`（任意で：難易度、紐づく単元名）。
- 依頼者に目次を手書きさせない（Voice Study と同じ思想。ファイル名・フォルダ構成から自動で本棚を組む）。
- 規則から外れたファイルを検知する**ファイル名チェッカー**を用意し、放り込むだけ運用を安全に回す。

## 6. 開発フロー（Claude Code / Codex 両対応）

依頼者は **Claude Code（仕事）と Codex（プライベート）両方**を使う。どちらでも動くよう設計する。

- 自宅PCの**ドライブG直下に新規プロジェクトフォルダ** `sukima/` を作る（Voice Study `G:\voice_study\` とは別）。
- `AGENTS.md` に「`CLAUDE.md` を読め」と書く（Codex対応）。`CLAUDE.md` に**改変不可の事項**を記述（**150行未満厳守**）。
- 設計書・仕様書は別 md に分け、**依頼者の許可があれば編集可**。
- 作業内容を毎回記録。`/clear` 推奨のキリのよいタイミングで**コミット・プッシュ**。
- 作業中の知見（失敗含む）を記録し、必要なら **skills や hooks を作成**して自律的に成長させる。
- **大タスク終了時に各 md の肥大化をチェックし、コンテキスト節約のため最適化**。
- バックエンド/フロントエンドの整合性チェックやタスク漏れ防止のため、**サブエージェントでの並列作業を許可**。

### CLAUDE.md（150行未満）に必ず入れる不変ルール
- 毎回この md を読め／小タスクごとにコミット・プッシュ／大タスクごとに md 最適化
- skills・hooks 作成の方針／作業ごとに記録せよ
- **本プロジェクト固有の歯止め**：
  - やることは3択クイズ1つだけ。やり込ませる機能（時間記録・グラフ・連続日数・ランキング）は作らない
  - GitHub Pages の静的アプリで完結させる（Cloudflare・サーバー処理を持ち込まない）
  - 正解＝回答済で消える／不正解＝未回答で残る／全問回答済でリセット可、の出題ロジックを守る
  - 問題は1単元1JSON・命名規則準拠。依頼者に目次を手書きさせない
  - 問題追加が最優先価値。追加が面倒になる作りにしない

## 7. 初期構築でエージェントに最初にやらせること

1. ドライブG直下に新規フォルダ `sukima/` を作成、`git init`。
2. GitHub に新規リポジトリを作成（依頼者の `sslazio29-lab` 系アカウント想定・要確認）、GitHub Pages の公開設定を通す。
3. 骨組みファイル一式を生成：`AGENTS.md`（→CLAUDE.md参照）、`CLAUDE.md`（150行未満の不変ルール）、設計書md（許可制編集）、記録用md。
4. 問題JSONのスキーマ（§5の1問の構造）を確定し、サンプル問題セットを1単元ぶん作る。
5. 静的アプリの最小版を立ち上げ、GitHub Pages へのデプロイ経路を通す（push→公開を確認）。
6. 出題テンポ（§3）の最小実装：科目→単元→問題数選択→3択→〇×と解説→次へ、正解/不正解の状態保存、リセット。

> いずれも設計書・仕様書は**依頼者の承認を得てから**着手する流れを踏襲すること。
> 依頼者は「以前フォルダ構成を自分で作って失敗した」経験から、初期ファイル群の生成はエージェントに任せる方針。
