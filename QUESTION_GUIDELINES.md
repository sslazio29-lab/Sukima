# QUESTION_GUIDELINES.md（作問・修正ルール）

Sukima の問題を作るときの実務ルール。`CLAUDE.md` の固定方針を前提に、問題品質の判断を補う。

## 基本方針

- 目的は、受験生がスキマ時間に「知識を思い出す」こと。難問化より、頻出知識の確実な定着を優先する。
- 問題は必ず3択。解説は、正解理由だけでなく「どこで誤りやすいか」を短く入れる。
- 1問1知識を基本にする。複数知識を同時に問う場合は、解説で判断手順を明示する。

## 選択肢の作り方

- 不正解2つも、同じ科目・同じ単元内の近接概念、よくある誤訳、紛らわしい分類を優先する。
- 「別時代すぎる語」「駄洒落」「文字数・ページ番号・作者の好き嫌い」など、知識がなくても切れる選択肢は避ける。
- ただし、初学者向けの単語問題では現代語との混同を利用してよい。その場合も露骨な冗談ではなく、実際に誤訳しやすい選択肢にする。
- 正解が複数に見える近接選択肢は避ける。迷わせるのはよいが、解説を読めば一つに定まること。
- 選択肢に「だけ」を使う場合は、安直な外れに見えないか確認する。

## 古文で特に注意すること

- 古語・助動詞・助詞・敬語は、訳語だけでなく、接続・文脈・敬意の方向まで説明する。
- 古典常識は、平安貴族社会・仏教観・宮中制度・装束・年中行事など、本文理解に効く知識へ寄せる。
- 文学史は、時代・ジャンル・作者・作品名の対応を問う。作品の特徴や受験での見分けどころを解説に入れる。
- 読みが問題になる語は、国語・古文で一般的な読みを優先する。例：`牛車` は `ぎっしゃ`。

## 修正時の確認

- 修正後は `npm.cmd run check` と `npm.cmd run audit` を必ず通す。
- 選択肢の重複、正答番号のズレ、説明と選択肢の不一致を確認する。
- push 後は、変更した単元を `npm.cmd run pages:check -- 科目 単元` で確認する。
