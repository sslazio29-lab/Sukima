> このファイルは **依頼者の承認があれば編集可**。`CLAUDE.md`（不変ルール）の範囲内で詳細を詰める。
> 初回構想セッションで確定した内容を、実装に落とせる粒度で記述する。

---

## 1. アーキテクチャ（確定）

純粋な静的アプリ。**GitHub Pages 一本**で完結する。

- 画面・判定ロジック・問題データ(JSON)・自動生成された `manifest.json` を1リポジトリに入れて公開。
- サーバー処理ゼロ。「正解はどれか」「解説表示」「回答済の記録」はすべて受験生の iPhone のブラウザ内。
- 問題JSONを一度読み込めば以降ほぼ通信しない。音声を配信しないため 3GB制約・オフライン保存の悩みは無関係。
- 端末：iPhone がメイン。同じURLを Surface の大画面で開くことも可能（静的なので自然に両対応）。

---

## 2. リポジトリ公開範囲（確定：public）

- **public リポジトリ＝完全無料**で進める。
- 理由：
  - 無料個人プランでは private リポジトリから GitHub Pages を公開できない（Pro 等の有料が必要）。
  - 有料で private にしても、**公開される Pages サイト経由で問題JSONはURL所持者から取得可能**。
    private が隠すのはソース履歴であって配信内容ではない。秘匿効果がコストに見合わない。
  - 家庭の学習教材であり、URL は推測困難。実害はほぼない（引き継ぎ書の判断を踏襲）。
- **重要な誤解の解消**：回答済記録は localStorage（端末内）にのみ保存。第三者が同じURLで解いても
  受験生の進捗には一切影響しない。public でも進捗共有は起きない。

## 3. データ構造（確定）

### ファイル単位
- **1ファイル＝1単元**。ファイル名 = **`科目_単元.json`**（例：`日本史_室町時代.json`、`英単語_ターゲット01.json`）。
- 連番は使わない。科目・単元は**ファイル名が唯一の正**。中身には科目・単元を書かない（二重管理による不一致を防ぐ）。

### 1問の構造
ファイルの中身は問題オブジェクトの**配列**。

```json
[
  {
    "問題文": "足利義満が始めた日明貿易の別名はどれか。",
    "選択肢": ["勘合貿易", "南蛮貿易", "朱印船貿易"],
    "正解": 1,
    "解説": "正式な遣明船と倭寇を区別するため勘合符を用いた。"
  }
]
```

- `選択肢` は必ず3つ。
- `正解` は **1〜3 の番号**（`選択肢` 配列の何番目が正解か、1始まり）。文字列一致より挙動が安定。
- 難易度フィールドは持たない。

### id の扱い（追加の手軽さの心臓部）
- **明示の id を持たせない。配列の並び順そのものが id**。
- 問題数 = 配列の長さ。ファイルを開けば今何問あるか一目で分かる。
- **追加は末尾に足すだけ**。末尾追加なら既存の問題の位置がズレず、進捗記録も壊れない。
- 途中挿入・並べ替えは位置（=id）をズラし進捗が崩れるので**避ける**。運用ルールは「足すなら末尾へ」の一点。

## 4. manifest 自動生成（確定：GitHub Actions）

静的サイトはブラウザからフォルダ一覧を取得できないため、存在する問題ファイルの目次
`manifest.json` が必要。これを手書きさせない。

- **push をトリガーに GitHub Actions が問題フォルダを走査**し、`manifest.json` を自動生成。
- 同時にファイル名規則（`科目_単元.json`）と JSON の妥当性（配列・選択肢3つ・正解が1〜3 等）を**検証**。
- 規則違反・壊れた JSON があれば **Action が失敗（赤いバツ）して通知**。無言の失敗を作らない。
- manifest からアプリが「科目→単元」の本棚を自動で組む。依頼者の作業は「JSONを末尾に足して push」だけ。
- 実装メモ：生成スクリプトは Node か Python の小さな1ファイルで十分。CI を一切持ちたくない場合のみ、
  エージェントがコミット前にローカルで生成する代替も可だが、手で足したときの検証漏れを招くため Actions を推奨。

## 5. 出題ロジック（確定）

- 各問は **未回答／回答済** の2状態。出題は未回答プールから引く。
- **正解 → 回答済（以後出ない）**。**不正解 → 未回答のまま（また引かれる）**。
- これだけで苦手な問題が自然にしつこく出る。**復習用の独立画面は作らない**（出題ロジックに溶かし込む）。
- 出題順は素直なシャッフルのみ。「最低 N 問あける」等の規則は足さない。
- **リセットは単元単位**。その単元が全問回答済になったらリセットボタンで未回答へ戻せる。他単元・他科目には触れない。

### localStorage キー設計
- 粒度は `科目/単元/問番号`（問番号=配列の位置）。例キー：`sukima:日本史:室町時代:answered`（回答済位置の集合）。
- 単元リセット＝その単元のキーのみ削除。他は不変。
- 保存先 API（localStorage 等）の最新挙動は実装直前に確認すること。

## 6. 画面フロー（極限までシンプル）

1. 科目を選ぶ → 単元を選ぶ
2. 問題数を選ぶ（受験生が毎回決める。固定しない。電車1区間・休み時間に終わる量を自分で調整）
3. 問題が出る → 選択肢3つをタップ
4. 即座に〇× → 解説 → 次へ
5. （その単元が全問回答済なら）リセット可

採点・タイマー・点数集計・演出は作らない。テンポだけを追求。

## 7. 科目展開の順序

1. 作りやすい暗記系から：**倫理・日本史・英単語**（Claude が量産）。
2. 並行して **数学**（依頼者が誤答を改題、解法選択型）。
3. その後、化学・生物・古文・漢文へ同じ器で拡張。

## 8. 画像対応の検討項目（未実装）

図形・グラフ・回路・化学構造など、画像がないと成立しにくい問題に限り、将来的に画像表示を検討する。
ただし、Sukima の基本は3択クイズであり、画像対応を標準運用にはしない。

### 方針
- GitHub Pages の静的配信内で完結させる。サーバー処理や外部ストレージは追加しない。
- 既存問題を壊さないため、問題JSONには任意フィールドとして `問題画像` / `解説画像` を追加する案を基本にする。
- 画像形式はまず SVG を優先する。小画面で拡大しても劣化せず、図形・グラフ・回路との相性がよい。
- 画像は「図がないと解けない、または理解が大きく落ちる問題」だけに使う。暗記・語彙・句法・方針選択は原則テキストのまま。
- iPhone SE3 の4.7インチ画面を前提に、図・選択肢・解説を一画面へ詰め込まない。縦スクロールを基本にし、必要なら画像タップで拡大表示する。
- manifest/check では、画像フィールドがある場合に参照先ファイルの存在を検証し、リンク切れを GitHub Actions で検出できるようにする。

### 保留
- KaTeX などによる本格的な数式組版は、画像対応とは別判断とする。インライン表記で読めなくなってから検討する。

## 9. 初期構築タスク（エージェント最初の着手・各ステップ承認制）

1. ドライブG直下に新規フォルダ `sukima/` を作成、`git init`。
2. GitHub に新規 public リポジトリ作成（`sslazio29-lab` 系アカウント想定・要確認）、GitHub Pages 公開設定を通す。
3. 骨組み生成：`AGENTS.md`・`CLAUDE.md`・`SPEC.md`・`WORKLOG.md`。
4. 問題JSONスキーマ（§3）を確定し、サンプル1単元ぶんを作成。
5. manifest 自動生成の GitHub Actions（§4）を用意し、push→生成→公開を確認。
6. 静的アプリ最小版：科目→単元→問題数→3択→〇×と解説→次へ／正解・不正解の状態保存／単元リセット。

> 依頼者は「以前フォルダ構成を自分で作って失敗した」経験から、初期ファイル群の生成はエージェントに任せる方針。
> いずれも仕様確定・着手前に依頼者の承認を得る。
